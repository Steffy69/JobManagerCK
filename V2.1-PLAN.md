## Confirmed Decisions

1. **Status model**: Three states — `Ready` → `In Progress` → `Printed`
   - Ready: no actions taken
   - In Progress: transferred / printed / nc_copied (at least one)
   - Printed: job folder moved to `S:\Jobs\Printed\` via "Move to Printed" button
   - Backwards compat: keep `completed_at` JSON field name
2. **Widget**: Replace `QListWidget` with `QTreeWidget` for collapsible sections
3. **Settings path**: `~/.jobmanager/settings.json` via `os.path.expanduser` (NOT hardcoded to `continental`)
4. **Printed jobs scan**: Full scan using same pipeline as active jobs (reuse `scan_folder_files`)
5. **Restore to Active button**: Included, but ONLY visible when a printed job is selected (hidden otherwise)
6. **Archive→Printed folder migration**: Auto-migrate on first launch. If `S:\Jobs\Archive\` exists and `S:\Jobs\Printed\` doesn't, move contents.
7. **Label printer routing**: Use `win32api.ShellExecute(0, "printto", file, f'"{printer_name}"', ...)` — targets Zebra directly, independent of Windows default printer

```

```

Two separator types:
- **Job separator** — first separator peeled, shows "JOB_NAME / FIRST_MATERIAL"
- **Material separator** — shows "MATERIAL" only

## Filename Format (discovered)

`JOBNAME_MATERIAL_0000.ljd`
Example: `JELPREWIR_WHMR_0001.ljd`

Parsing: split by `_`, material = middle segment, board number = last segment (strip leading zeros).

Verified from: `S:\Jobs\Custom Design\JELPREWIR CL\Labels\`
Materials observed: WHMR (34), WALNUT (16), BlackHMR (3)

## Print Order Logic

### Rule (peel vs print)
- **Peel order** = what Marinko pulls off the roll in sequence
- **Print order** = reverse of peel order (last printed = top of stack = first peeled)

### Within material
Reverse numeric board order (34 → 33 → ... → 1) so board 1 ends up on top when the user starts peeling that material group.

### Between materials
User's priority list = **peel order** (top of list = peeled first). Print engine reverses for print order.

Default for unlisted materials: sort by board count descending (largest first in peel order) — so bulk material (WHMR, usually) is always on top.

### Separator positions
For each material M in print order, print M's labels, then append a separator:
- If M is the LAST material in print order (= first peeled, topmost in stack): job separator (JOB_NAME / M)
- Otherwise: material separator (M only)

### Example for JELPREWIR CL (default priority)

Peel order desired:
```
1. [SEP] JELPREWIR CL — WHMR
2-35.    WHMR 1 → 34
36. [SEP] WALNUT
37-52.   WALNUT 1 → 16
53. [SEP] BlackHMR
54-56.   BlackHMR 1 → 3
```

Actual print sequence:
```
1-3:   BlackHMR 3, 2, 1       (deepest material group in stack)
4:     [SEP] BlackHMR
5-20:  WALNUT 16, 15, ..., 1
21:    [SEP] WALNUT
22-55: WHMR 34, 33, ..., 1
56:    [SEP] JELPREWIR CL — WHMR   (last printed = top of stack)
```

Total: 56 prints (53 labels + 3 separators)

## Risks

| Risk | Severity | Mitigation |
|---|---|---|
| Status rename collision breaks user mental model | Resolved | Three distinct states |
| QTreeWidget refactor misses a reference → runtime crash | High | Grep for `jobListWidget`, smoke test all operations |
| Print sequencing logic bug (wrong order on roll) | High | Extensive unit tests with known expected sequences, Marinko manual verification |
| pywin32 + PyInstaller bundling fails | Medium | Explicit hiddenimports, test clean-directory launch |
| `ShellExecute("printto")` doesn't route to Zebra | Medium | Fallback: SetDefaultPrinter before print loop |
| ZPL separator doesn't print correctly on GC420D | Medium | Test with simple ZPL first, Marinko verifies |
| Archive→Printed migration fails mid-move | Medium | Check both folders exist after move; log errors loudly |
| Settings dialog drag-reorder feels awkward | Low | Standard QListWidget with Qt.InternalMove drag mode |
| Preflight checks slow down operations | Low | Use short-timeout checks only |
| Clear print queue needs admin | Medium | Catch error, show friendly message, document powershell fallback |
